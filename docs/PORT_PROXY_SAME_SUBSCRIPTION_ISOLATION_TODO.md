# 记录：同一订阅多端口代理的端口级运行视图与选择状态隔离

## 当前状态

已按最终确认方案实现端口级运行视图：

- 节点页会按 `订阅名(端口代理名称)` 展示端口代理运行视图。
- 后端会按端口代理实例生成端口级 `proxy-groups` 副本。
- 每个 listener 的 `sub-rules` 会指向自己的端口级代理组。
- 订阅自带 `sub-rules` 也会按端口复制，`SUB-RULE` 引用会改写到端口级子规则名。
- 真实 `proxies` 与 `proxy-providers` 不按端口复制，只复用订阅级前缀后的名称。
- 同一订阅被多个端口代理使用时，`select` 代理组选择状态按端口代理隔离。
- 端口级代理组的 `now` 优先从对应 `port_proxies[].selected` 恢复；旧数据才回退读取兼容层 `profiles.current.selected`，不要把它理解成用户侧“主订阅”。

## 背景问题

端口代理支持多个端口同时绑定订阅。如果两个端口代理绑定同一个订阅，并且订阅中存在同名代理组，例如：

```yaml
proxy-groups:
  - name: 节点选择
    type: select
    proxies:
      - 美国01
      - 日本01
      - 新加坡01
```

如果最终运行配置中两个端口都引用同一份 `节点选择` 代理组，那么 Mihomo 内部只有一份选择状态：

```text
节点选择 = 美国01
```

这样无法做到：

```text
10001 -> 节点选择 -> 美国01
10002 -> 节点选择 -> 日本01
```

根因不是端口监听，也不是节点复用，而是 `proxy-groups` 的选择状态按代理组名称保存；同名同组天然共享当前选中节点。

## 最终方案

把订阅运行视图从“订阅级”提升到“端口代理级”。

显示名前缀：

```text
订阅名(端口代理名称) - 原名称
```

示例：

```text
iKuuu_V2(工作端口) - 节点选择
iKuuu_V2(游戏端口) - 节点选择
```

这样 Mihomo 会认为它们是两个不同的代理组，从而拥有独立选择状态。

## 运行配置示例

```yaml
listeners:
  - name: 工作端口
    type: mixed
    listen: 127.0.0.1
    port: 10001
    rule: port-proxy-work

  - name: 游戏端口
    type: mixed
    listen: 127.0.0.1
    port: 10002
    rule: port-proxy-game

proxy-groups:
  - name: iKuuu_V2(工作端口) - 节点选择
    type: select
    proxies:
      - iKuuu_V2 - 美国01
      - iKuuu_V2 - 日本01

  - name: iKuuu_V2(游戏端口) - 节点选择
    type: select
    proxies:
      - iKuuu_V2 - 美国01
      - iKuuu_V2 - 日本01

sub-rules:
  port-proxy-work:
    - DOMAIN-SUFFIX,google.com,iKuuu_V2(工作端口) - 节点选择
    - MATCH,iKuuu_V2(工作端口) - 节点选择

  port-proxy-game:
    - DOMAIN-SUFFIX,google.com,iKuuu_V2(游戏端口) - 节点选择
    - MATCH,iKuuu_V2(游戏端口) - 节点选择
```

说明：

- 端口级副本只复制代理组与规则引用。
- 真实节点仍复用订阅级节点名称，避免复制大量节点配置。
- 跨订阅合并时节点名称仍使用 `订阅名 - 原名称` 前缀来避免跨订阅冲突。
- 系统代理 / TUN 的运行订阅当前通过同步到原版 `profiles.current` 兼容原增强流程；端口代理则按端口实例生成独立运行视图。

## 前端行为

### 节点页

节点页顶部会同时展示：

- 未被开启端口代理引用的启用订阅视图，例如 `iKuuu_V2`；
- 已开启端口代理运行视图，例如 `iKuuu_V2(工作端口)`、`iKuuu_V2(游戏端口)`。

如果某个订阅已经被开启端口代理引用，节点页不再同时显示一个原订阅视图，避免出现：

```text
iKuuu_V2
iKuuu_V2(工作端口)
```

这种重复入口。也就是说，一个订阅只被一个端口代理使用时，节点页只显示一个订阅页签，且页签名显示为原订阅名；同一订阅被多个端口代理使用时，才显示多个 `订阅名(端口代理名称)` 端口运行视图。

节点页页签顺序按订阅页的订阅顺序展开：未被端口代理引用的订阅显示原订阅视图；被端口代理引用的订阅在原位置替换为对应端口运行视图，不再把“使用了的订阅”统一追加到最后。

端口代理运行视图只在端口代理模式实际生效时参与节点页展示。系统代理 / TUN 模式下，即使配置文件里残留开启状态的端口代理，也不应由端口视图遮蔽系统代理 / TUN 正在使用的订阅视图。

节点页中的代理组标题、右侧代理组导航、链式代理规则组选择菜单只做展示层去前缀：把 `订阅名(端口代理名称) - 原代理组名` 显示为 `原代理组名`。内部真实名称不变，避免破坏 Mihomo 运行时选择状态隔离。

系统代理 / TUN 后续也应纳入同一“运行订阅实例”表达：它们各自选择一个启用订阅，但同一时间只会有一个入口模式实际运行。

当多个端口代理选择同一订阅时，会出现多个端口级视图：

```text
iKuuu_V2(工作端口)
iKuuu_V2(游戏端口)
```

用户在不同端口级视图中选择节点，会写入不同的运行时代理组名，例如：

```json
{ "name": "iKuuu_V2(工作端口) - 节点选择", "now": "iKuuu_V2 - 美国01" }
{ "name": "iKuuu_V2(游戏端口) - 节点选择", "now": "iKuuu_V2 - 日本01" }
```

### 端口代理页

端口代理卡片继续显示：

- 订阅；
- 最近命中的节点组；
- 最近命中的节点；
- 端口级链式代理状态。

展开端口代理时，可以在当前端口代理对应的节点组中选择节点；选择动作会作用到该端口代理自己的端口级代理组。

## 链式代理关系

端口代理有两种路径：

```text
不开链式代理：listener -> rule -> port sub-rules -> 端口级代理组 -> 节点
开链式代理：listener -> proxy -> 端口级链式出口
```

因此：

- 普通端口代理按订阅 rules / sub-rules 分流。
- 端口级链式代理是端口级全局出口链，不先经过订阅规则命中节点组。
- 链式节点仍使用当前端口代理绑定订阅的名称映射，避免引用错节点。

## 验证用例

### 用例 1：同订阅双端口不同节点

1. 启用订阅 `iKuuu_V2`。
2. 创建端口代理 A：端口 `10001`，绑定 `iKuuu_V2`。
3. 创建端口代理 B：端口 `10002`，绑定 `iKuuu_V2`。
4. 在 `iKuuu_V2(端口代理 A)` 视图中选择 `美国01`。
5. 在 `iKuuu_V2(端口代理 B)` 视图中选择 `日本01`。
6. 分别通过两个端口访问同一测试地址。
7. 确认两个端口实际出口不同。

### 用例 2：同订阅双端口相同节点

1. A 和 B 都选择 `美国01`。
2. 确认两个端口都可访问，互不占用、互不影响。

### 用例 3：关闭其中一个端口

1. 关闭 A。
2. 确认 `10001` 不再监听。
3. 确认 `10002` 仍可正常访问。
4. 确认运行配置里 A 对应 listener 不再存在。

### 用例 4：端口级链式代理

1. A 开启链式代理。
2. B 不开启链式代理。
3. 确认 A 走链式出口，B 仍按订阅规则分流。

## 注意事项

- 不要把端口级前缀写回原订阅文件；前缀只出现在最终运行配置中。
- 如果端口代理名称变化，端口级运行组名称也会变化；旧选择状态不会继续命中，需要重新选择节点。
- 如果订阅更新导致已保存节点不存在，运行时会回退到代理组自身默认 `now` 或首个可用项。
- provider 类型订阅需要同时处理 `use`、`proxy-providers`、`rule-providers` 和 `RULE-SET` 引用。
